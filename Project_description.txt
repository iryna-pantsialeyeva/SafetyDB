Проект "Safety database"

На сайте фармкомпании (далее - Заказчик) необходимо предусмотреть возможность коммуникации с пользователями продукции компании (препаратов).
Поскольку наиболее часто на сайт Заказчика пользователи приходят, чтобы пожаловаться и сообщить о побочных реакциях на препараты Заказчика, 
далее будем называть их "репортерами" (=человек, составляющий репорт).
Репортерами могут быть медработники, пациенты или их родители (=типы репортеров).

Попадая на сайт Заказчика репортеру предлагается выбор следующих действий:
"Просим выбрать необходимое действие:
1. Сообщить о побочной реакции на препарат;
2. Получить список побочных реакций на препарат;
3. Удалить отправленный ошибочный репорт;
4. Ввести обновленные данные для уже ранее поданного репорта;
5. Выход из программы

Pепортер делает выбор, вводя цифру, соответствующую своему выбору. Если вводит цифру, которой нет в списке действий, 
ему предлагается сделать выбор повторно.

1. save - "Сообщить о побочной реакции на препарат"
1.1. Репортеру предлагается заполнить поля Формы репорта, куда он вводит следующие данные (названия полей: термин на русяз -> название переменной в IDEA -> название поля и таблицы в sql БД)
- название побочной реакции -> description -> description VARCHAR(255) / table adverse_reactions
- название подозреваемого препарата - suspectedDrug -> suspected_drug VARCHAR(255) / table adverse_reactions
- критерий серьезности (степень серьезности побочной реакции) criteria  -> name VARCHAR(255) / table seriousness_criterias
  (id из seriousness_criterias присваивается полю criteria_id в table adverse_reactions)
- исход побочной реакции (чем закончилась побочная реакция) outcome -> name VARCHAR(255) / table outcomes
  (id из outcomes присваивается полю outcome_id в table adverse_reactions)
- ФИО репортера - fullName -> full_name VARCHAR(255) / table reporters
  (id из reporters присваивается полю reporter_id в table adverse_reactions)
- тип репортера (кем является заполняющий форму репорта - медработник или просто пациент) type -> name VARCHAR(255) / table reporter_types
  (id из reporter_types присваивается полю reporter_type_id в table adverse_reactions)
- уникальный номер побочной реакции (генерируется автоматически) -> id -> id INT / table adverse_reactions
- дата репорта (генерируется автоматически) -> reportDate -> report_date DATE / table adverse_reactions
*Все поля в форме репорта являются обязательными для заполнения репортером. Если он не заполняет любое из полей, сразу, 
без перехода на другое поле выводится предупреждение "Данное поле является обязательным для заполнения. Пожалуйста введите данные."
1.2. Выявление дублей в БД
До сохранения репорта проводится проверка на выявление дублей репортов в БД (есть ли такой репорт в БД). 
Дублем считается совпадение полей suspectedDrug и description (*В реальной жизни такая проверка проводится по бОльшему количеству полей репорта).
Если указанные поля совпадают, выводится предупреждение "Репорт не может быть принят. Данный репорт уже есть в БД."
Если указанные поля не совпадают, репорт сохраняется в sql БД и выводится запись "Ваш репорт успешно загружен в БД." 

2. get - "Получить список побочных реакций на препарат"
2.1. При выборе данной опции репортеру предлагается ввести название препарата, по которому он хотел бы получить данные из БД.
2.2. Если название препарата не соответствует ни одной записи поля suspectedDrug в БД, выводится сообщение "Для данного препарата в БД отсутствуют репорты."
В противном случае - выводится список репортов, содержащий в поле suspectedDrug название препарата, введенное репортером. 

3. delete - "Удалить отправленный ошибочный репорт"
3.1. При выборе данной опции репортеру предлагается ввести свои ФИО и дату сделанного им репорта, который он планирует удалить.
Проводится поиск репортов, у которых обя поля fullName и reportDate соответствуют тем, которые введены репортером. 
Если таких репортов нет, выводится сообщение "В БД отстутствуют репорты, сообществующие Вашему запросу."
В противном случае - удаляется репорт и репортеру сообщается о том, что "Ошибочный репорт успешно удален."

4. update - "Ввести обновленные данные для уже ранее поданного репорта"
4.1. При выборе данной опции репортеру предлагается ввести свои ФИО и дату сделанного им репорта, который он планирует удалить.
Проводится поиск репортов, у которых обя поля fullName и reportDate соответствуют тем, которые введены репортером. 
Если таких репортов нет, выводится сообщение "В БД отстутствуют репорты, сообществующие Вашему запросу."
В противном случае - репортеру предлагается обновить данные в имеющемся репорте. Поля, доступные для обновления - description, suspectedDrug, criteria, outcome. 

5.  Выход из программы - программа прощается с репортером "Благодарим за то, что воспользовались формой обратной связи. Мы работаем ради вашей благополучия..." (что-то такое).
